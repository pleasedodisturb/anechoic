# GitLab Handbook - Comprehensive Summary

**Source:** [handbook.gitlab.com](https://handbook.gitlab.com)  
**Company:** GitLab (All-Remote, ~1200+ employees)  
**Handbook Size:** 2000+ pages  
**Format:** Summarized into core bulletpoints per section

## Company Philosophy

### Handbook-First Approach
- **Every process, decision, and policy documented before implementation**
- Company operates with "public by default" philosophy
- Restricted only for security, legal, or privacy reasons
- Demonstrates all-remote operations at 1000+ employee scale
- Handbook is the single source of truth - if it's not in the handbook, it doesn't exist

### Values (CREDIT)
- **Collaboration** - Help others succeed
- **Results** - Focus on outcomes, not activity
- **Efficiency** - Work smarter, not harder
- **Diversity, Inclusion & Belonging** - Everyone can contribute
- **Iteration** - Ship minimal viable changes
- **Transparency** - Default to public

## Remote Work Manifesto

### All-Remote Core Principles
- **Default Async Communication:** Real-time is exception, not rule
- **Written First:** Documentation is primary communication method
- **Searchable Information:** All decisions and discussions archived and findable
- **Results Over Activity:** Focus on outcomes, not hours or presence
- **Location Independence:** Work from anywhere without penalty
- **No Hybrid:** Fully remote, not partially (avoids two-tier system)

### Why All-Remote Works
- Access to global talent pool
- No commute = better work-life balance
- Forced documentation improves clarity
- Asynchronous = respects everyone's time
- Written communication = more thoughtful decisions

## Communication Protocols

### Asynchronous Communication
- **Default to async** - synchronous requires justification
- **24-48 hour response expectation** for non-urgent matters
- **Write it down** - if it's not documented, it didn't happen
- **Low-context communication** - assume reader has no background
- **Over-communicate** - repeat key information across channels

### When to Use Sync
- Sensitive personnel matters
- Urgent incidents/outages
- Complex negotiations
- Relationship building (occasional)
- Onboarding first week

### Meeting Guidelines
- **No agenda, no meeting** - agenda required 24h in advance
- **Meetings are expensive** - calculate cost before scheduling
- **Record everything** - so async attendees can catch up
- **Start on time, end early** - respect everyone's calendar
- **Document outcomes** - meeting notes posted within 24h

### Communication Channels
| Channel | Purpose | Response Time |
|---------|---------|---------------|
| GitLab Issues | Work discussions, decisions | 24-48h |
| Slack | Quick questions, social | 24h |
| Email | External, formal | 48h |
| Video calls | When sync is justified | Scheduled |

## Documentation Practices

### Handbook-First
- **Document first, discuss later** - create MR before meeting
- **Anyone can contribute** - handbook accepts merge requests
- **Living document** - continuously updated, never "done"
- **Searchable** - organized for discoverability
- **Canonical** - single source of truth for any topic

### Documentation Standards
- Write for someone with no context
- Include the "why" not just "what"
- Link to related pages
- Date and attribute changes
- Use clear headings and structure

### Decision Documentation
- **RFC (Request for Comments)** for major decisions
- Document alternatives considered
- Record who decided and when
- Link to implementation
- Update when decisions change

## Hiring & Onboarding

### Remote-First Hiring
- All roles open to global candidates unless location-specific
- Assessment focuses on culture fit and value alignment
- Preference for candidates with varied experiences
- Job descriptions include compensation ranges, benefits
- Async interview options available

### Onboarding
- **Buddy System:** Each new hire paired with mentor for first 30 days
- **Handbook is onboarding:** Primary resource for new hires
- **Role-Specific Guides:** Department handbooks available
- **First-Week Structure:** Clear outline of expectations
- **Onboarding issues:** Checklist of tasks to complete

### Performance & Development
- **Result-Focused Evaluation:** Measured on output, not activity
- **Quarterly Reviews:** Structured feedback cycles
- **Career Paths:** Transparent promotion criteria
- **Continuous Learning:** Budget allocated for development

## Team Practices

### 1:1 Meetings
- Weekly or bi-weekly with manager
- Employee sets agenda
- Focus on career development, not status
- Document action items
- Async pre-work expected

### Team Meetings
- Monthly all-hands (recorded)
- Team syncs as needed (always recorded)
- Social calls optional but encouraged
- No mandatory attendance (watch recording)

### Working Hours
- No set hours - results matter
- Communicate availability in calendar
- Respect others' time zones
- Core collaboration windows optional per team
- Family and personal time protected

## Key Takeaways for TPMs

1. **Document everything** - Create a program handbook
2. **Default to async** - Challenge every meeting request
3. **Write for no context** - Assume readers are new
4. **Record all sync** - Enable async participation
5. **Measure results** - Not activity or presence
6. **Iterate publicly** - Share work early and often
