## Card 1: Async Message Writing

### The Perfect Async Message

```
✅ CONTEXT: What's the situation?
✅ REQUEST: What do you need?
✅ DEADLINE: When do you need it?
✅ FORMAT: How should it be delivered?
✅ LINKS: All references included
```

### Example

❌ **Bad:**
> "Can you send me reporting when you get a chance?"

✅ **Good:**
> "Can you send me a 2-paragraph summary of blog metrics from Google Analytics by **Tuesday 2pm EST**? Need: top posts, page views, bounce rate, conversion rate. Here's the dashboard: [link]"

### Quick Tips
- One message, all context
- Assume reader has no prior knowledge
- Use bullet points for clarity
- Include screenshots when helpful
- Specify timezone for deadlines

## Card 2: Meeting Audit

### Before Scheduling, Ask:

```
□ Can this be a document?
□ Can this be an async thread?
□ Can this be a Loom video?
□ Is there a decision to make that requires sync?
□ Do we need rapport-building?
```

### If You Must Meet:

| Requirement | Action |
|-------------|--------|
| Agenda | Create editable doc, share 24h before |
| Notes | Assign note-taker OR everyone contributes |
| Duration | Use 25 or 50 minutes (not 30/60) |
| Recording | Record for async attendees |
| Follow-up | Post summary to thread within 1 hour |

### Scripts to Decline

> "Could we try to solve this in a thread first?"

> "I'd love to give feedback! Could I review in a doc before we schedule?"

> "Let me draft something async and we can meet only if needed."

## Card 3: Documentation Habits

### The Handbook-First Flow

```
1. WRITE the answer/process/decision
2. PUBLISH to the handbook/wiki
3. SHARE link (not content) to relevant channels
4. UPDATE when things change
```

### What to Document

| Always Document | Never Document |
|-----------------|----------------|
| Decisions made | Casual chat |
| Processes | Temporary info |
| Meeting outcomes | Personal opinions |
| How-to guides | Work-in-progress ideas |
| Policies | Confidential HR matters |

### Where Things Live

| Type | Location |
|------|----------|
| Permanent knowledge | Handbook |
| Active discussions | GitHub Issues/PRs |
| Quick questions | Slack (public channels) |
| Decisions | #decision-records |
| Meeting notes | Linked doc + thread |

## Card 4: Response Time Guidelines

### Setting Expectations

| Channel | Expected Response | Use For |
|---------|-------------------|---------|
| Handbook link | Self-service | FAQs, processes |
| Async thread | Within 24 hours | Normal work |
| Slack mention | Same day (work hours) | Awareness |
| DM (avoid!) | — | Personal only |
| Emergency channel | Immediate | True crises |

### Your Response Checklist

```
□ Acknowledge receipt if can't answer now
□ Give deadline: "I'll have this by Thursday"
□ Don't say "will do" - say when
□ If quick task: just do it now
□ If blocked: say what's blocking
```

### The 24-Hour Rule

> "At Doist, we expect people to answer within 24 hours."

## Card 6: Slack Hygiene

### Do's ✅

- Use **public channels** for work discussions
- **Thread** all replies
- **React** with ✅ when question is answered
- One message with all context
- Include links to everything referenced

### Don'ts ❌

- DMs for work topics (move to public)
- Multiple messages for one thought
- @channel/@here unless truly urgent
- Storing decisions (use docs/handbook)
- Expecting immediate responses

### Channel Naming

```
#team-[name]      → Team discussions
#project-[name]   → Cross-team projects
#ask-[topic]      → Questions
#announce-[area]  → One-way updates
#social-[topic]   → Non-work chat
```

## Card 7: The DRI Model

### Direct Responsible Individual

> One person owns every decision/project. Others advise, DRI decides.

### Benefits

- **Speed:** No consensus needed
- **Clarity:** Everyone knows who decides
- **Accountability:** Clear ownership
- **Async-friendly:** Decisions don't wait for meetings

### How It Works

```
1. ASSIGN a DRI for each project/area
2. DRI GATHERS input asynchronously
3. DRI DECIDES (doesn't need consensus)
4. DRI DOCUMENTS the decision
5. DRI COMMUNICATES to stakeholders
```

### When to Escalate

- Decision is irreversible
- Significant budget impact
- Major customer impact
- Conflict with company values

## Card 8: Building Connection Remotely

### Scheduled Connection Rituals

| Ritual | Frequency | Purpose |
|--------|-----------|---------|
| **1:1s** | Weekly | Manager-report relationship |
| **Fika** | Ad-hoc | Cross-team connection |
| **Team Social** | Monthly | Team bonding |
| **All-Hands** | Weekly/Monthly | Company alignment |
| **Retreats** | Annual | Deep relationship building |

### Async Connection Ideas

- **Personal check-ins** in team standups (red/yellow/green)
- **#social channels** for non-work chat
- **Donut/random pairing** for cross-team intros
- **Celebrations** channel for wins
- **Show & tell** async videos

### The 70/25/5 Rule (Doist)

```
70% → Async (threads, docs, issues)
25% → Sync (video calls, meetings)
 5% → In-person (retreats, offsites)
```

## Card 9: Writing for Async

### Structure for Clarity

```markdown
## TL;DR
[One sentence summary]

## Context
[Background needed to understand]

## Proposal/Request
[What you're suggesting or need]

## Options Considered
[Alternatives and trade-offs]

## Recommendation
[Your suggested path forward]

## Next Steps
[What happens after this]
```

### Writing Tips

1. **Lead with the ask** - Don't bury the point
2. **Use headers** - Scannable structure
3. **Bullet points** - Not walls of text
4. **Bold key points** - Guide the eye
5. **Link liberally** - Context without repetition
6. **Edit ruthlessly** - Shorter is better

### Avoid

- Jargon and acronyms
- Passive voice
- "Let's sync on this" (say what you need)
- "ASAP" (give real deadline)
- Emotional language in text

## Card 10: Emergency Escalation

### What is a Real Emergency?

✅ **Emergency:**
- System outage affecting customers
- Security breach
- Legal/compliance deadline
- Safety issue

❌ **Not Emergency:**
- Need answer for meeting tomorrow
- Stakeholder is anxious
- Someone else said it's urgent
- You forgot to plan ahead

### Emergency Protocol

```
1. POST in #emergency channel (notifications ON)
2. CALL if no response in 15 min
3. DOCUMENT incident as it happens
4. POST-MORTEM within 48 hours
```

### Preventing Fake Urgency

> "Most things aren't urgent, but making them seem so will add a higher tax of anxiety." — Doist

**Ask:**
- What happens if this waits 24 hours?
- Is this my emergency or someone else's?
- Did poor planning create this urgency?
